# Project Messiah v2 Plan

## Objective

Maximize the probability of **first revenue** and **repeatable profit**.

That means we optimize for:

- narrow scope,
- fast delivery,
- clear acceptance criteria,
- low refund / rejection risk,
- and outcomes a human buyer can understand immediately.

## Core shift

Do not start as a general autonomous bounty hunter.

Start as a **human-sold, agent-assisted delivery engine** for narrow, high-clarity tasks.

The agent helps us move faster, but the commercial promise is:

**"You get a clearly scoped result fast, with proof."**

## Revenue model

We want three revenue layers, in this order:

1. fixed-price services,
2. recurring retainers,
3. one reusable paid API.

Bounties and agent marketplaces are supporting channels, not the center.

## Core product stack

The product assumptions are now:

- LangGraph is the workflow engine,
- Groq is the primary fast inference provider,
- PySide6 is the local operator dashboard,
- and coding work is only one service line among several.

This means Messiah is an orchestration product, not just a marketplace strategy.

## Expanded market thesis

Messiah should not be a "coding jobs only" system.

It should target any work that has these properties:

- clear input,
- clear output,
- clear acceptance criteria,
- low legal risk,
- repeatable process,
- and high time-value to the buyer.

That creates a much wider service map than software engineering alone.

## Service lines to target

We should think in service lines, not job titles.

## 1. Technical delivery

This is still one of the best lines because proof is objective.

Examples:

- bug fixes
- CI / deploy recovery
- API integration repair
- small automations
- code reviews
- AI-output QA

## 2. Operations and admin execution

This is often less competitive than coding and easier to standardize.

Examples:

- spreadsheet cleanup and restructuring
- CRM data cleanup
- lead list enrichment
- inbox triage
- SOP formatting
- document conversion and cleanup
- data entry with validation
- weekly reporting packs

## 3. Research and synthesis

This can be valuable when tightly scoped and evidence-based.

Examples:

- competitor research
- tool comparison matrices
- market landscape briefs
- grant / investor list research
- vendor shortlists
- policy or regulation summaries

Rule:

Sell research only when the output format is explicit.

## 4. Content production with structure

Do not sell "creative genius." Sell structured business content.

Examples:

- landing page rewrites
- email sequences
- outreach personalization batches
- SEO content briefs
- product descriptions
- FAQ generation from source materials

## 5. Sales enablement

This is attractive because buyers tie it directly to pipeline growth.

Examples:

- lead qualification packs
- ICP lists
- sales research dossiers
- proposal drafting
- follow-up sequences

## 6. Support and customer success operations

Good fit when the workflow is repetitive and policy-bound.

Examples:

- help-center drafting
- ticket classification
- canned response libraries
- escalation summaries
- churn-risk notes from call transcripts

## 7. Agent-facing microservices

This remains an optional expansion path after human demand is proven.

Examples:

- document cleanup API
- structured research API
- classification API
- code QA API
- repo context packaging API

## What not to chase

Even with a wider market, we should still avoid categories with bad risk-adjusted economics.

- work requiring licenses we do not have
- medical or legal decision-making
- subjective creative commissions with vague acceptance criteria
- highly custom consulting with no repeatable process
- manual-heavy low-ticket work with no automation edge
- speculative finance / trading promises
- jobs requiring deception, fake identities, or policy evasion

## Exact offers to sell first

These are the first offers I would sell across multiple lanes.

## 1. Bug Rescue

### Promise

Fix one reproducible bug in an existing codebase.

### Good fit

- auth bugs,
- API integration failures,
- Stripe / webhook failures,
- CI or deployment breakage,
- UI regression with clear repro steps,
- Supabase / database wiring issues.

### Deliverables

- root cause summary,
- code fix,
- test or validation note,
- deployment / merge guidance.

### Suggested pricing

- starter: $99 for trivial reproducible bug
- standard: $249 for normal app bug
- premium: $499 for tricky multi-file bug

### Why this offer works

- buyers understand it,
- pain is immediate,
- scope can be constrained,
- and acceptance is easy to verify.

## 2. Codebase Health Audit

### Promise

Review one repository and produce a prioritized action report.

### Deliverables

- top 5 technical risks,
- quick wins,
- dependency and CI notes,
- optional implementation quote.

### Suggested pricing

- audit only: $79
- audit + action plan: $149

### Why this works

This is a low-friction entry product that feeds higher-value repair work.

## 3. CI / Deployment Recovery

### Promise

Get a failing pipeline or broken deploy back to green.

### Deliverables

- diagnosis,
- fix or config patch,
- steps to verify,
- prevention notes.

### Suggested pricing

- $149 to $399 depending on stack and access.

### Why this works

Deployment pain is urgent and budget is usually available.

## 4. Automation Sprint

### Promise

Build one small automation that saves repeated manual work.

### Good fit

- form to spreadsheet,
- email triage,
- CRM sync,
- Slack alerts,
- scheduled report generation,
- small internal dashboard.

### Suggested pricing

- $199 to $599 fixed scope.

### Why this works

It is easier to sell savings than speculative AI magic.

## 5. AI Output QA

### Promise

Review and harden AI-generated code before it ships.

### Deliverables

- defect list,
- risk notes,
- optional patch pass.

### Suggested pricing

- review only: $49 to $99
- review + repair: $149 to $299

### Why this works

There is growing distrust of raw AI output, which creates demand for a trusted final pass.

## 6. Research Brief Sprint

### Promise

Produce one tightly scoped research pack with sources and a decision-ready summary.

### Good fit

- competitor maps
- tool comparisons
- vendor shortlists
- market scans
- grant or investor lists

### Deliverables

- summary memo,
- comparison table,
- source links,
- recommendation or shortlist.

### Suggested pricing

- $79 to $249 depending on depth.

### Why this works

Many buyers need clarity, not genius. A structured brief is easier to standardize than open-ended consulting.

## 7. Data Cleanup Sprint

### Promise

Clean, normalize, deduplicate, or restructure one messy business dataset.

### Good fit

- CRM exports
- spreadsheet cleanup
- category normalization
- duplicate record removal
- lead list validation

### Deliverables

- cleaned file,
- rules used,
- issue summary,
- optional repeatable workflow note.

### Suggested pricing

- $49 to $199 depending on size and messiness.

### Why this works

This is painful, frequent work with clear before/after proof.

## 8. Content System Pack

### Promise

Turn rough notes or source material into business-ready content assets.

### Good fit

- landing page copy
- email sequences
- help-center drafts
- FAQ systems
- product page rewrites

### Deliverables

- finished copy assets,
- source-to-output mapping,
- revision note.

### Suggested pricing

- $59 to $249 depending on asset count.

### Why this works

Structured content is easier to QA than custom creative work.

## Offers to avoid initially

- custom "build my whole app" projects
- vague research gigs
- security bounty automation
- crypto trading automation as a service
- open-ended assistant retainers
- multi-week builds with unclear specs

These destroy scope control.

## Platform order

As of March 24, 2026, I would target platforms in this order.

## Tier 1: Best starting channels

### 1. Upwork Project Catalog

Why first:

- pre-scoped offers fit our model,
- clients can buy without long proposal cycles,
- and Upwork says Project Catalog has over 290,000 projects.

Use it for:

- Bug Rescue
- CI Recovery
- Codebase Audit
- Research Brief Sprint
- Data Cleanup Sprint

## 2. Contra Services

Why second:

- service-menu format,
- commission-free positioning,
- and strong fit for productized offers.

Use it for:

- Codebase Audit
- Automation Sprint
- Content System Pack
- Research Brief Sprint
- fractional follow-up work

## 3. Fiverr

Why third:

- good for simple, narrow, repeatable offers,
- but heavy marketplace noise means we should keep scope very tight.

Use it for:

- Bug Rescue Lite
- AI Output QA
- CI Fix Review
- Data Cleanup Sprint
- FAQ / content packs

## Tier 2: Controlled experimental channels

### 4. Claw Earn

Why fourth:

- good payment rails,
- single-start structure reduces duplicate work,
- but it is still beta and crypto-native.

Use it for:

- fast, bounded tasks under 100 USDC early on,
- especially tasks with objective deliverables.
- avoid ambiguous creative or advisory tasks.

### 5. Bounty.new

Why fifth:

- real platform with docs and payment flow,
- but still early-stage and competitive by design.

Use it for:

- issues that are clearly scoped and under-contested,
- only after we have stronger proof of delivery quality.

## Tier 3: Non-platform channels

These may outperform marketplaces once the offer is working:

- direct outreach to startup founders
- outreach to indie hackers shipping SaaS products
- partnerships with freelancers and small agencies
- GitHub issue-to-offer outreach on repos with obvious broken flows

This is where margin often improves.

## Positioning

The brand should not sound like a robot bounty spammer.

It should sound like:

**fast, scoped delivery with disciplined execution**

Good positioning lines:

- "Small, scoped work shipped fast."
- "We turn messy work into finished deliverables."
- "Human-reviewed, AI-accelerated execution."
- "Clear scope. Clear proof. Fast turnaround."

Bad positioning lines:

- "fully autonomous revenue agent"
- "AI generalist for anything"
- "crypto-native bounty intelligence layer"

Those lines increase skepticism.

## Operational rules

These rules matter more than clever architecture.

1. Never accept vague work.
2. Never promise a timeframe without repro steps.
3. Never auto-submit external work without human review.
4. Prefer objective acceptance criteria.
5. Log time, cost, and outcome for every job.
6. Build repeatable delivery templates.

## Messiah v2 product architecture

We are not building a giant autonomous platform first.

We are building a **delivery operating system** for small, structured jobs.

## Runtime architecture

### LangGraph

LangGraph should coordinate the job lifecycle:

- intake
- qualification
- estimation
- execution planning
- evidence collection
- delivery generation

### Groq

Groq should handle the low-latency model calls for:

- classification
- summarization
- first-pass planning
- extraction tasks
- structured content generation

### PySide6 dashboard

PySide6 should remain in the project as the local operator dashboard.

The dashboard should show:

- inbound queue
- service line
- estimated value
- confidence
- approval status
- delivery artifacts
- cost and turnaround metrics

The dashboard is not decorative. It is the human control plane.

## Modules

### `intake`

Normalizes inbound jobs from forms, platform listings, or fixtures.

### `qualification`

Rejects vague or risky jobs and extracts acceptance criteria.

### `estimation`

Scores difficulty, confidence, time, and likely margin.

### `execution_plan`

Breaks approved work into concrete steps.

### `evidence`

Collects logs, notes, file changes, and proof of completion.

### `delivery`

Formats the buyer-facing output: summary, deliverables, validation, next steps.

### `metrics`

Tracks win rate, margin, turnaround time, and refund / rejection patterns.

### `dashboard`

Provides the PySide6 control surface for queue review, approvals, and metrics.

## What the first version should do

Input:

- a local JSON job brief

Output:

- qualification decision,
- service line,
- estimated scope,
- price band,
- execution plan,
- delivery checklist.

That gives us a working dry-run commercial engine.

The first UI layer should also be able to display those outputs locally in the PySide6 dashboard.

## First file to write next

The next file should be `src/messiah/models.py`.

Why:

- every LangGraph node will depend on shared types,
- the dashboard needs typed objects to display,
- and multi-service qualification requires richer models than a coding-only agent.

## Decision

If the goal is money with the highest probability, Messiah should be:

**a productized scoped-work business powered by internal agent workflows**

not

**an autonomous bounty empire**
